# Experiments

## Методика

Эксперименты выполняются командой:

```bash
make deps
make up
make migrate
make experiment
make verify-results
```

Скрипт использует фиксированный seed `4116`, сбрасывает состояние стенда через `/admin/reset`, выполняет сценарии в заданном порядке и сохраняет результаты в `results/latest/`. Производительная часть запускается сериями `100/500/1000` операций, по 5 повторов для сценариев «синхронная транзакция», «Outbox-транзакция» и «чатовая рассылка». Сценарии внедрения отказов выполняются контролируемыми одиночными запусками, потому что их цель — не пропускная способность, а проверка конкретного инварианта.

## Сценарии

| Сценарий | Проверка | Ожидаемый результат |
| --- | --- | --- |
| `sync_transaction`, синхронная транзакция | Серии синхронных команд начисления золота. | p50/p95/p99, пропускная способность и доля ошибок зафиксированы, команды изменили данные в PostgreSQL. |
| `outbox_transaction`, Outbox-транзакция | Серии команд с записью Outbox и асинхронной публикацией. | События доставлены фоновым обработчиком, остаток Outbox отсутствует после ожидания. |
| `chat_fanout`, чатовая рассылка | Серии чатовых сообщений. | Сообщения записаны в PostgreSQL и Redis Streams, задержка и пропускная способность зафиксированы. |
| `duplicate_callback`, повторный callback | 5 одинаковых команд с одним `message_id`. | Баланс изменён один раз, 4 запроса распознаны как дубли. |
| `publish_no_ack`, сбой после публикации | Фоновый обработчик публикует событие, но не отмечает Outbox доставленным. | Redis Streams содержит транспортный дубль, Outbox в итоге закрыт. |
| `pubsub_loss`, потеря Pub/Sub-сообщения | Сообщение публикуется до подписки. | Поздний подписчик его не получает. |
| `cache_stale`, устаревшее чтение кэша | Значение читается из cache-aside, затем PostgreSQL меняется без инвалидации. | Повторное чтение возвращает устаревший кэш. |
| `saga`, Saga | Одна успешная и одна компенсированная покупка. | Наблюдаются `saga_completed >= 1` и `saga_compensated >= 1`. |
| `websocket_replay`, WebSocket-восстановление | WebSocket-клиент подключается с `last=0-0`. | Клиент получает хотя бы одно событие при повторном чтении. |

## Артефакты

| Артефакт | Назначение |
| --- | --- |

| `comparison_matrix.csv` | Итоговые оценки решений по критериям, рассчитанные из метрик и сценариев внедрения отказов. |

## Проверка результатов

`scripts/verify_results.py` проверяет следующие условия:

- выполнены 9 агрегированных серий и 45 отдельных запусков нагрузки;
- размеры нагрузки равны `100/500/1000`, число повторов равно 5;
- задержка, пропускная способность и доля ошибок измерены для каждого класса обработки;
- Inbox предотвратил повторное изменение состояния;
- транспортный дубль после `publish_no_ack` обнаружен;
- Pub/Sub потерял сообщение для позднего подписчика;
- cache-aside вернул устаревшее значение;
- saga завершилась и компенсировалась;
- WebSocket-восстановление доставило события;
- после эксперимента нет остатка Outbox.
- созданы все CSV/JSON/PNG-артефакты;
- трассировка доказательств содержит строки по масштабируемости, надёжности, производительности, сложности внедрения, стоимости эксплуатации и наблюдаемости.

Если хотя бы один инвариант нарушен, команда `make verify-results` завершается с ненулевым кодом.

## Интерпретация

Стенд доказывает не абсолютную пригодность конкретных чисел задержки для промышленного кластера, а причинные свойства решений. Например, `publish_no_ack` обязан дать транспортный дубль, потому что публикация и отметка Outbox не атомарны относительно Redis. Это не опровергает Outbox, а показывает, что Outbox должен применяться вместе с идемпотентными потребителями. Аналогично, потеря сообщения поздним Pub/Sub-подписчиком не является ошибкой Redis; это проверка того, что Pub/Sub нельзя использовать как единственный гарантированный канал критичного состояния.
