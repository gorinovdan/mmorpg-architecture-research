# Experiments

## Методика

Эксперименты выполняются командой:

```bash
make deps
make up
make migrate
make experiment
make verify-results
```

Скрипт использует фиксированный seed `4116`, сбрасывает состояние стенда через `/admin/reset`, выполняет сценарии в заданном порядке и сохраняет результаты в `results/latest/`. Производительная часть запускается сериями `100/500/1000` операций, по 5 повторов для каждого из сценариев `sync_transaction`, `outbox_transaction` и `chat_fanout`. Fault-сценарии выполняются контролируемыми одиночными запусками, потому что их цель — не throughput, а проверка конкретного инварианта.

## Сценарии

| Сценарий | Проверка | Ожидаемый результат |
| --- | --- | --- |
| `sync_transaction` | Серии синхронных команд начисления золота. | p50/p95/p99, throughput и error rate зафиксированы, команды изменили PostgreSQL. |
| `outbox_transaction` | Серии команд с записью Outbox и асинхронной публикацией. | События доставлены worker, pending Outbox отсутствует после ожидания. |
| `chat_fanout` | Серии чатовых сообщений. | Сообщения записаны в PostgreSQL и Redis Streams, latency и throughput зафиксированы. |
| `duplicate_callback` | 5 одинаковых команд с одним `message_id`. | Баланс изменен один раз, 4 запроса распознаны как дубли. |
| `publish_no_ack` | Worker публикует событие, но не отмечает Outbox доставленным. | Redis Streams содержит транспортный дубль, Outbox в итоге закрыт. |
| `pubsub_loss` | Сообщение публикуется до подписки. | Поздний подписчик его не получает. |
| `cache_stale` | Значение читается из cache-aside, затем PostgreSQL меняется без инвалидации. | Повторное чтение возвращает устаревший кэш. |
| `saga` | Одна успешная и одна компенсированная покупка. | Наблюдаются `saga_completed >= 1` и `saga_compensated >= 1`. |
| `websocket_replay` | WebSocket-клиент подключается с `last=0-0`. | Клиент получает хотя бы одно replay-событие. |

## Артефакты

| Артефакт | Назначение |
| --- | --- |
| `summary.json` | Полная сводка запуска: параметры, агрегаты, fault-сценарии, финальные метрики, матрица сравнения, evidence trace и рекомендации. |
| `metrics.csv` | Нормализованная метрика в формате `scenario, metric, value, unit`. |
| `scenario_runs.csv` | Все 45 запусков серий нагрузки. |
| `comparison_matrix.csv` | Итоговые оценки решений по критериям, рассчитанные из метрик и fault-сценариев. |
| `criteria_scores.csv` | Развернутая форма матрицы для построения графиков. |
| `evidence_trace.json` | Цепочка `критерий -> сценарий -> метрика -> артефакт -> результат -> вывод`. |
| `recommendations.json` | Практические рекомендации с указанием доказательства. |

## Проверка результатов

`scripts/verify_results.py` проверяет следующие условия:

- выполнены 9 агрегированных серий и 45 отдельных запусков нагрузки;
- размеры нагрузки равны `100/500/1000`, число повторов равно 5;
- latency, throughput и error rate измерены для каждого класса обработки;
- Inbox предотвратил повторное изменение состояния;
- транспортный дубль после `publish_no_ack` обнаружен;
- Pub/Sub потерял сообщение для позднего подписчика;
- cache-aside вернул stale value;
- saga завершилась и компенсировалась;
- WebSocket replay доставил события;
- после эксперимента нет pending Outbox.
- созданы все CSV/JSON/PNG-артефакты;
- evidence trace содержит строки по масштабируемости, надежности, производительности, сложности внедрения, стоимости эксплуатации и наблюдаемости.

Если хотя бы один инвариант нарушен, команда `make verify-results` завершается с ненулевым кодом.

## Интерпретация

Стенд доказывает не абсолютную пригодность конкретных чисел latency для промышленного кластера, а причинные свойства решений. Например, `publish_no_ack` обязан дать транспортный дубль, потому что публикация и отметка Outbox не атомарны относительно Redis. Это не опровергает Outbox, а показывает, что Outbox должен применяться вместе с идемпотентными потребителями. Аналогично, потеря сообщения поздним Pub/Sub-подписчиком не является ошибкой Redis; это проверка того, что Pub/Sub нельзя использовать как единственный durable-канал критичного состояния.
