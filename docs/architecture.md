# Architecture Notes

## Границы эксперимента

Стенд моделирует минимальную серверную часть игрового сервиса:

- команды игрока изменяют баланс золота;
- чатовые и доменные события публикуются наружу;
- WebSocket-клиент восстанавливает события после повторного подключения;
- purchase saga либо завершается, либо компенсируется, а время выполнения фиксируется через `created_at` и `finished_at`;
- cache-aside показывает риск устаревших данных при отсутствии строгой инвалидации.

В стенде нет полноценного игрового движка, шардирования мира, матчмейкинга и авторизации. Эти компоненты исключены намеренно: исследование проверяет не игровую механику, а архитектурные свойства интеграционных решений.

## Компоненты

| Компонент | Назначение |
| --- | --- |
| HTTP-шлюз | Принимает команды `/command`, `/chat`, `/saga/purchase` и административные запросы эксперимента. |
| WebSocket-шлюз | Отдает историю из Redis Streams и затем оперативные события из Pub/Sub. |
| PostgreSQL | Хранит достоверные данные состояния игроков, Inbox, Outbox, сообщений и Saga. |
| Redis Streams | Сохраняемый журнал событий для повторного чтения и проверки дублированных транспортных событий. |
| Redis Pub/Sub | Низколатентная краткоживущая доставка оперативных событий. |
| Фоновый обработчик | Обрабатывает Outbox и Saga, публикует события в Redis. |
| Python-сценарий | Выполняет нагрузочные сценарии и сценарии внедрения отказов, сохраняет метрики. |

## Наблюдаемость

Метод `/metrics/summary` агрегирует состояние PostgreSQL и Redis. Он предназначен не для имитации промышленной точки сбора метрик Prometheus, а для доказательной связи между экспериментом и выводом. В сводку включены:

| Метрика | Проверяемый вывод |
| --- | --- |
| `inbox_count` | callback и команды фиксируются в Inbox, поэтому можно проверить факт дедупликации. |
| `outbox_published`, `outbox_attempts_total` | процесс публикации Outbox реально публикует события, а повторные попытки наблюдаемы. |
| `outbox_pending`, `outbox_oldest_pending_ms` | после серии нагрузок не остается незакрытой очереди Outbox. |
| `redis_stream_len`, `redis_duplicate_events` | Redis Streams хранит историю и позволяет обнаружить транспортный дубль. |
| `chat_messages`, `chat_stream_len` | чатовые сообщения проходят через сохраняемый журнал, а не только через Pub/Sub. |
| `saga_completed`, `saga_compensated`, `saga_duration_avg_ms`, `saga_duration_p95_ms` | Saga имеет наблюдаемые исходы и измеряемую длительность. |

## Потоки данных

### Синхронная транзакция

1. Клиент отправляет команду.
2. HTTP-шлюз открывает транзакцию PostgreSQL.
3. Inbox фиксирует `message_id`.
4. Баланс игрока изменяется в таблице `players`.
5. Команда завершается после фиксации транзакции.

Такой режим показывает нижнюю границу сложности: состояние надежно изменено, но внешнее событие не отделено от бизнес-транзакции.

### Outbox/Inbox

1. Команда записывается в Inbox.
2. Изменение игрока и запись Outbox выполняются в одной транзакции.
3. Фоновый обработчик читает непоставленные Outbox-события.
4. Событие публикуется в Redis Streams и Pub/Sub.
5. Outbox помечается как доставленный.

При сценарии `publish_no_ack` фоновый обработчик публикует событие, но не отмечает запись доставленной. Следующая попытка публикует тот же `event_id` повторно. Это воспроизводит классический сбой «публикация выполнена, подтверждение не зафиксировано» и доказывает необходимость идемпотентных потребителей.

### Redis Pub/Sub и Streams

Pub/Sub используется только как краткоживущий оперативный канал. Поздний подписчик не получает старое сообщение, что фиксируется сценарием `pubsub-loss`. Redis Streams хранит события и позволяет WebSocket-клиенту запросить повторное чтение после повторного подключения.

### Saga

Saga purchase имеет два проверяемых исхода:

- `completed`: у игрока достаточно золота и предмет доступен;
- `compensated`: предмет отсутствует или средств недостаточно.

Сценарий показывает, что распределенное действие не следует сводить к одной локальной транзакции. Для MMORPG это важно при покупке предметов, начислении наград, торговле и взаимодействии с инвентарем.

В таблице `sagas` хранятся `created_at` и `finished_at`. Это позволяет проверять не только факт завершения или компенсации, но и длительность Saga как эксплуатационную метрику.

### Cache-aside

Endpoint `/player?cache=true` сначала читает Redis, затем PostgreSQL. Эксперимент намеренно изменяет PostgreSQL в обход инвалидации и повторно читает значение из кэша. Это демонстрирует риск устаревшего чтения, который должен учитываться в рекомендациях по TTL, инвалидации и выбору данных, допустимых к кэшированию.

## Инварианты

- Повторная доставка одной команды не меняет состояние повторно.
- Outbox не должен оставаться с недоставленными событиями после завершения эксперимента.
- Outbox может породить транспортный дубль при `publish_no_ack`; это считается ожидаемым сигналом, а не ошибкой, если потребитель идемпотентен и очередь закрыта.
- Redis Streams должен сохранять события для WebSocket-восстановления.
- Redis Pub/Sub не должен рассматриваться как гарантированный журнал критичных событий.
- Saga должна иметь явно наблюдаемый успешный и компенсированный исход.
- Устаревшее чтение кэша должно обнаруживаться отдельным проверочным сценарием.
