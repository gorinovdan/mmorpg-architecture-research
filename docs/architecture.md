# Architecture Notes

## Границы эксперимента

Стенд моделирует минимальный backend игрового сервиса:

- команды игрока изменяют баланс золота;
- чатовые и доменные события публикуются наружу;
- WebSocket-клиент восстанавливает события после reconnect;
- purchase saga либо завершается, либо компенсируется;
- cache-aside показывает риск устаревших данных при отсутствии строгой инвалидации.

В стенде нет полноценного игрового движка, шардирования мира, матчмейкинга и авторизации. Эти компоненты исключены намеренно: исследование проверяет не игровую механику, а архитектурные свойства интеграционных решений.

## Компоненты

| Компонент | Назначение |
| --- | --- |
| HTTP gateway | Принимает команды `/command`, `/chat`, `/saga/purchase` и административные запросы эксперимента. |
| WebSocket gateway | Отдает историю из Redis Streams и затем live-события из Pub/Sub. |
| PostgreSQL | Source of truth для состояния игроков, Inbox, Outbox, сообщений и Saga. |
| Redis Streams | Durable event log для replay и проверки дублированных транспортных событий. |
| Redis Pub/Sub | Низколатентная эфемерная доставка live-событий. |
| Worker | Обрабатывает Outbox и Saga, публикует события в Redis. |
| Python runner | Выполняет нагрузочные и fault-injection сценарии, сохраняет метрики. |

## Потоки данных

### Синхронная транзакция

1. Клиент отправляет команду.
2. Gateway открывает транзакцию PostgreSQL.
3. Inbox фиксирует `message_id`.
4. Баланс игрока изменяется в таблице `players`.
5. Команда завершается после commit.

Такой режим показывает нижнюю границу сложности: состояние надежно изменено, но внешнее событие не отделено от бизнес-транзакции.

### Outbox/Inbox

1. Команда записывается в Inbox.
2. Изменение игрока и запись Outbox выполняются в одной транзакции.
3. Worker читает непоставленные Outbox-события.
4. Событие публикуется в Redis Streams и Pub/Sub.
5. Outbox помечается как delivered.

При fault `publish_no_ack` worker публикует событие, но не отмечает запись доставленной. Следующая попытка публикует тот же `event_id` повторно. Это воспроизводит классический сбой "publish succeeded, ack failed" и доказывает необходимость идемпотентных потребителей.

### Redis Pub/Sub и Streams

Pub/Sub используется только как live-канал. Поздний подписчик не получает старое сообщение, что фиксируется сценарием `pubsub-loss`. Redis Streams хранит события и позволяет WebSocket-клиенту запросить replay после reconnect.

### Saga

Saga purchase имеет два проверяемых исхода:

- `completed`: у игрока достаточно золота и предмет доступен;
- `compensated`: предмет отсутствует или средств недостаточно.

Сценарий показывает, что распределенное действие не следует сводить к одной локальной транзакции. Для MMORPG это важно при покупке предметов, начислении наград, торговле и взаимодействии с инвентарем.

### Cache-aside

Endpoint `/player?cache=true` сначала читает Redis, затем PostgreSQL. Эксперимент намеренно изменяет PostgreSQL в обход инвалидации и повторно читает cached value. Это демонстрирует риск stale read, который должен учитываться в рекомендациях по TTL, инвалидации и выбору данных, допустимых к кэшированию.

## Инварианты

- Повторная доставка одной команды не меняет состояние повторно.
- Outbox не должен оставаться с pending-событиями после завершения эксперимента.
- Redis Streams должен сохранять replay-события для WebSocket reconnect.
- Redis Pub/Sub не должен рассматриваться как durable event log.
- Saga должна иметь явно наблюдаемый успешный и компенсированный исход.
- Stale cache должен обнаруживаться автоматической проверкой.
