# MMORPG Architecture Research Stand

Исследовательский стенд проверяет архитектурные решения серверной платформы текстовой MMORPG: хранение достоверного игрового состояния в PostgreSQL, Inbox/Outbox, Redis Streams/Pub/Sub, WebSocket-восстановление, Saga-оркестрацию и cache-aside. Репозиторий предназначен для воспроизводимого запуска экспериментов, результаты которых используются в отчете НИР.

Публичный URL: <https://github.com/gorinovdan/mmorpg-architecture-research>

## Состав стенда

| Компонент | Назначение |
| --- | --- |
| Серверная часть на Go | HTTP API, WebSocket-шлюз, обработка команд, чат, фоновая обработка Saga и административные сценарии эксперимента. |
| PostgreSQL | Хранит достоверное состояние игроков, Inbox, Outbox, сообщения чата и состояния Saga. |
| Redis Streams | Сохраняемый журнал событий для повторного чтения, учета дублей и восстановления WebSocket-клиента после повторного подключения. |
| Redis Pub/Sub | Низколатентный краткоживущий канал оперативной рассылки активным клиентам. |
| Python-сценарий | Серии нагрузки, сценарии внедрения отказов, CSV/JSON-артефакты и графики. |
| GitHub Actions | Go-тесты, проверка Docker Compose, генерация и проверка экспериментальных артефактов. |

## Быстрый запуск

```bash
make deps
make test
make up
make migrate
make experiment
make verify-results
make down
```

`make deps` создает локальное Python-окружение `.venv` и устанавливает зависимости для построения графиков. `make experiment` использует фиксированный seed `4116`, выполняет серии нагрузки и сценарии внедрения отказов, затем сохраняет результаты в `results/latest/`.

## Экспериментальная методика

Стенд строит сравнение не на экспертной оценке, а на цепочке:

`критерий -> сценарий стенда -> метрика -> результат -> вывод -> рекомендация`.

Проверяемые критерии:

- масштабируемость: серии нагрузки `100/500/1000` для `sync_transaction`, `outbox_transaction` и `chat_fanout`, по 5 повторов;
- надежность: повторный callback, сбой после публикации, потеря Pub/Sub-сообщения для позднего подписчика, WebSocket-восстановление, устаревшее чтение кэша, успешная и компенсированная Saga;
- производительность: p50/p95/p99 задержки, пропускная способность и доля ошибок;
- сложность внедрения: число реализованных компонентов, состояний, контрактов и путей фоновой обработки;
- стоимость эксплуатации: компоненты с собственным состоянием, длина журнала, повторные попытки, очередь необработанных событий, требования к истории;
- наблюдаемость: наличие метрик, по которым можно доказать или опровергнуть архитектурный вывод.

## Интерпретация результатов

Стенд не моделирует весь production-кластер MMORPG. Его назначение другое: воспроизвести классы отказов и проверить причинные свойства паттернов.

Основные подтверждаемые выводы:

- достоверное состояние игровой экономики, прогресса игрока, инвентаря игрока и аудита игровых операций хранится в PostgreSQL;
- Inbox и идемпотентность предотвращают повторный бизнес-эффект при повторной доставке callback;
- Transactional Outbox устраняет риск двойной записи между PostgreSQL и брокером, но требует идемпотентных потребителей;
- Redis Pub/Sub пригоден для оперативной рассылки активным клиентам, но не является сохраняемым журналом;
- Redis Streams нужен для повторного чтения и восстановления WebSocket-клиента;
- Saga оправдана только для долгих экономических цепочек с явными компенсациями;
- cache-aside не должен использоваться для решений о критичных игровых операциях из-за риска устаревшего чтения;
- архитектурная рекомендация считается доказанной только при наличии наблюдаемой метрики или сценария внедрения отказа.

## Ограничения валидности

Локальные значения задержки и пропускной способности не являются универсальным эталоном производительности. Они зависят от машины, Docker, версии среды выполнения и фоновой нагрузки. В отчете используются не абсолютные значения как промышленная норма, а воспроизводимая структура доказательств: инварианты, сценарии внедрения отказов, наличие очереди необработанных событий, повторная доставка, повторное чтение событий, исходы Saga и устаревшее чтение кэша.
